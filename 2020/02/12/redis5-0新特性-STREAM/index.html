redis5.0新特性-STREAM

发表于 2020-02-12 分类于 redis 阅读次数： Valine：
本文字数： 13k 阅读时长 ≈ 23 分钟

新增stream数据结构类型

stream与list等类型其实有大多数的相似功能，借鉴了kafka类似设计，是一个新的强大的支持多播的可持久化消息队列

比较项 Stream List, Pub/Sub, Zset
复杂度 获取元素高效，复杂度为O(logN) List获取元素的复杂度为O(N)
offset 支持offset，每个消息元素有唯一id。不会因为新元素加入或者其他元素淘汰而改变id。 List没有offset概念，如果有元素被逐出，无法确定最新的元素
持久化 支持消息元素持久化，可以保存到AOF和RDB中。 Pub/Sub不支持持久化消息。
消费分组 支持消费分组 Pub/Sub不支持消费分组
消息确认 支持ACK（消费确认） Pub/Sub不支持
性能 Stream性能与消费者数量无明显关系 Pub/Sub性能与客户端数量正相关
逐出 允许按时间线逐出历史数据，支持block，给予radix tree和listpack，内存开销少。 Zset不能重复添加相同元素，不支持逐出和block，内存开销大。
删除元素 不能从中间删除消息元素。 Zet支持删除任意元素

订阅一个Stream

  1. 一个流可以有多个客户端（消费者）在等待数据。默认情况下，每个新消息都将交付给每个正在等待给定流中数据的使用者。此行为与阻塞列表不同，在阻塞列表中，每个使用者都将获得不同的元素。然而，stream类似于Pub / Sub。
  2. 当在发布/订阅消息中时，消息会被下发并且删除，并且永远不会被存储；而在使用阻塞列表时，当客户端接收到一条消息时，该消息会从列表中弹出（有效删除），流的工作方式从根本上来说是不同的。所有消息将无限期地附加到流中（除非用户明确要求删除消息）：不同的使用者将通过记住收到的最后一条消息的ID，从其角度了解什么是新消息。
  3. 流使用者组提供了一个发布/订阅列表或阻止列表无法实现的控制级别，对于同一流，不同的组具有对处理流的显式确认、检查待处理项的能力，单一客户端具有对未处理消息的声明以及每个视图的连贯历史可见性，只能看到其过去的私人消息历史。

订阅一个Stream可以通过XREAD方式

相同的， 订阅不仅仅是单纯的直接获取消息，我们需要实时等待最新的消息的话可以使用BLOCK参数去阻塞获取

消费群体

在某些问题中，我们要做的不是向多个客户端提供相同的消息流，而是向同一客户端提供从同一流中不同的消息子集。一个服务中处理订阅消费内容，如果单一的服务器处理不过来， 就可以组件多台服务器，通过不同的服务器消费不同部分的消息从而进行负载均衡。这一点和kafka的消费群的实现概念一样，一个消费组就等于一个伪消费者，在同一个流上获取消息， 实际为多个使用者提供服务。

  1. 每个消息都提供给不同的使用者，因此不可能将同一消息传递给多个使用者。
  2. 消费者在消费者组中由名称标识，名称是区分大小写的字符串，实现消费者的客户端必须选择该字符串。这意味着，即使在断开连接后，流使用者组也将保留所有状态，因为客户端将再次声称自己是同一使用者。但是，这也意味着由客户端提供唯一的标识符。
  3. 每个使用者组都具有第一个从未使用过的ID的概念，因此，当使用者请求新消息时，它只能提供以前从未发送过的消息。
  4. 但是，使用一条消息需要使用特定命令来明确确认，例如：此消息已正确处理，因此可以从使用者组中逐出。
  5. 使用者组跟踪所有当前待处理的消息，即已传递给该使用者组的某个使用者但尚未被确认为已处理的消息。借助此功能，在访问流消息的历史记录时，每个使用者只会看到传递给它的消息。

建立消费群体

创建了使用者组，可以立即开始使用XREADGROUP命令尝试通过使用者组读取消息。

XREADGROUP基于我们指定的ID具有以下行为：

  • 如果该ID是特殊ID，>则该命令将仅返回到目前为止从未传递给其他使用者的新消息，并且，副作用是将更新使用者组的最后一个ID。
  • 如果该ID是任何其他有效的数字ID，则该命令将使我们能够访问待处理消息的历史记录。也就是说，传递给此指定使用者（由提供的名称标识）的消息集，到目前为止尚未使用XACK进行确认。

已经确认过的消息将不会再返回任何信息

当再次使用Bob用户获取消息的时候，返回的是第二条消息， 因为第一条已经被Alice使用了。以此类推，组中的任何使用者都可以从同一流中读取不同的消息，以读取其尚未处理的消息的历史记录，或将消息标记为已处理。

  • 消费者是在第一次被提及时自动创建的，无需显式创建。
  • 即使使用XREADGROUP，也可以同时读取多个键，但是要使此键起作用，您需要在每个流中创建一个具有相同名称的使用者组。这不是普遍的需求，但是值得一提的是该功能在技术上可用。
  • XREADGROUP是一种写命令，因为即使它从流中读取，使用者组也会被修改为读取的副作用，因此只能在主实例中调用它。

从故障中恢复消息

Redis使用者组提供了一种在这些情况下使用的功能，以便声明给定使用者的未决消息，以便此类消息将更改所有权并将其重新分配给其他使用者。该功能非常明确，使用者必须检查待处理消息列表，并且必须使用特殊命令声明特定的消息，否则服务器将永远将已分配的待处理消息分配给旧使用者，这样不同的应用程序可以选择是否使用此类功能，以及使用方式的确切选择。

通过添加不同的参数， 可以返回一些其他的内容

如果一个消息的空闲时间非常长（可以考虑为服务down掉）， 这个时候我们需要将消息取消分配权，并且分配给其他的使用者。

可以通过XINFO 命令查看STREAM中的信息

同时也可以查看group的信息

如果想更加细致的观察stream和group可以通过CONSUMERS参数

Kafka（TM）分区的差异

Redis流中的消费者组可能在某种程度上类似于基于Kafka（TM）分区的消费者组，但是请注意，Redis流实际上是非常不同的。分区仅是逻辑分区，消息仅放在单个Redis密钥中，因此为不同客户端提供服务的方式取决于谁准备处理新消息，而不是从客户端从哪个分区读取。例如，如果使用者C3在某个时刻永久失败，则Redis将继续为C1和C2提供所有到达的新消息，就像现在只有两个逻辑分区一样。
同样，如果给定的使用者处理消息的速度比其他使用者快得多，则该使用者将在相同的时间单位内按比例接收更多消息。这是可能的，因为Redis明确跟踪所有未确认的消息，并记住谁收到了哪个消息以及第一个消息的ID从未传递给任何使用者。
但是，这也意味着在Redis中，如果您确实要将同一个流中的消息划分为多个Redis实例，则必须使用多个密钥和某些分片系统，例如Redis Cluster或某些其他特定于应用程序的分片系统。单个Redis流不会自动分区到多个实例。
我们可以说，以下事实是正确的：

  • 如果您使用1个流-> 1个使用者，则按顺序处理消息。

  • 如果您对N个使用者使用N个流，以便只有给定的使用者命中N个流的子集，则可以缩放上述模型，即1个流-> 1个使用者。

  • 如果使用1个流-> N个使用者，则负载均衡到N个使用者，但是在那种情况下，有关同一逻辑项的消息可能会被无序使用，因为给定的使用者处理消息3的速度比另一个使用者处理的速度更快讯息4。

    因此，基本上，Kafka分区与使用N个不同的Redis密钥更相似。虽然Redis使用者组是一个服务器端负载平衡系统，用于将消息从给定流发送到N个不同的使用者。

封顶流

大多数情况下， 我们不需要将数据永久储存在stream中， 所以可以在达到一定的数量后将旧条目移出

当前没有选项告诉流只保留不超过给定数量的项目，因为这样的命令为了一致地运行，可能必须阻塞很长时间才能逐出项目。例如，想象一下，如果有一个插入峰值，然后是一个长时间的停顿（正在清除上一个时间峰值插入的数据），然后又有另一个插入，并且所有时间都相同，那么会发生什么。该流将阻塞以退出在暂停期间变得太旧的数据。因此，由用户来进行一些计划并了解所需的最大流长度是多少。 而且使用MAXLEN会消耗大量的资源， 因为需要对树进行修整， 所以可以使用 XADD mystream MAXLEN ~ 1000 * … entry fields here …， ~是说并不需要正好1000个， 可以是约等于这个数值，只要确保最少是1000个就好。仅在可以进行修整的时候进行剪枝处理以确保效率。

当然还可以通过XTRIM进行剪枝处理， 通过这个命令不需要添加任何消息

持久性，复制和消息安全

与任何其他Redis数据结构一样，Stream被异步复制到从属服务器并持久存储到AOF和RDB文件中。但是，可能不太明显的是，消费方组的完整状态也会传播到AOF，RDB和从属服务器，因此，如果主服务器中有待处理的消息，则从属服务器也将具有相同的信息。同样，重新启动后，AOF将还原使用者组状态。

  • 如果消息的持久性在您的应用程序中很重要，请将AOF刷盘方式调到最优。
  • 默认情况下，异步复制将不保证XADD命令或使用者组状态更改得以复制：故障转移后，根据从站从主站接收数据的能力，可能会丢失某些内容。
  • 使用WAIT命令可以以迫使这些变化的传播至一组从站使用。但是请注意，尽管这使得丢失数据的可能性很小，但由Sentinel或Redis Cluster操作的Redis故障转移过程仅尽最大努力检查故障转移到最新更新的从站，并且在某些特定故障下可能会导致故障。缺少一些数据的slave。

从流中删除单个项目

XDEL 可仅通过ID即可从流的中间删除项，删除一个消息并不一定会立马回收内存， 只有宏节点回收的时候才会回收内存

零长度流

在redis中， 一般当list或者set中最后一个元素被移出的时候而自动删除这个对象。由于使用MAXLEN选项（计数为零）（XADD和XTRIM命令）或由于调用了XDEL，因此流被保留为零元素，但是不会删除这个流对象。

之所以存在这种不对称性，是因为Streams可能具有关联的使用者组，并且我们不想因为使用者组中不再有项目而失去使用者组定义的状态。当前，即使没有关联的使用者组，流也不会被删除，但是将来可能会改变。
