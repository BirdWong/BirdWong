MySQL-复制

发表于 2019-10-11 分类于 MySQL 阅读次数： Valine：
本文字数： 15k 阅读时长 ≈ 27 分钟

MySQL从3.23开始提供复制功能，是指将主数据库的DDL和DML操作通过二进制日志传到复制数据库上，然后在从库上对这些日志重新执行，从而使得从库和主库的数据保持同步

MySQL支持一台主库同时向多态从库进行复制，从库同时可以作为其他服务器的主库，实现链状复制。如果主库出现问题，可以快速切换到从库提供服务；可以在从库上执行查询操作，降低主库的访问压力（因为复制是异步复制，所以对于实时性要求较高的服务还是要从主库中获取）；可以在从库上执行备份，以免备份期间影响主库的服务

复制概述

MySQL复制流程

  1. MySQL主库在事务提交时把数据变更作为时间Events记录在二进制日志文件中Binlog中；MySQL主库上的sync_binlog参数控制Binlog日志刷新到磁盘
  2. 主库推送二进制日志文件Binlog中的事件到从库中继日志Relay Log，之后从库更具中继日志Relay Log重新做数据变更操作，通过逻辑复制以此来达到主库和从库的数据一直

MySQL复制通过三个线程完成，Binlog Dump线程跑在主库上，I/O线程和SQL线程跑在从库上。当从库上启动Start Slave时，首先创建I/O线程连接主库，主库随后创建Binlog Dump线程读取数据库事件并发送给I/O线程，I/O线程获取到事件数据后更新到从库的中继日志Relay Log中去，之后从库上的SQL线程读取中继日志Relay Log中更新的数据库事件并应用

graph TB

D-->|4.SQL线程应用中继日志中的事件|A

B-->|DML Commands|C(BinLog)

A(Slave Database)-->|1.从库连接主库|B(Master Database)
C-->|3.Binlog dump线程读取数据库事件并把数据发给I/O线程|D(Relay Log)
A-->|2.I/O线程向主库要求数据|B

通过show processlist命令在主库上查看Binlog Dump线程，从Binlog Dump线程的状态可以看到，MySQL的复制是主库主动推送日志到从库去的，是属于推日志的方式来做同步。在从库中也可以看到I/O线程和SQL线程。

复制中的各个文件

MySQL复制流程中有两个非常重要的问及那：BinLog和RelayLog

BinLog会把MySQL中所有的数据修改操作记录到日志，RelayLog的内容与BinLog一致，但是SQL线程完成当前RelayLog中的事件之后会自动删除当前RelayLog，避免从库上的RelayLog占用过多磁盘

为保证从库Crash重启后，从库I/O仍然知道在哪开始复制，从库会创建两个日志文件master.info（I/O线程读取BinLog进度）和relay-log.info（SQL线程RelayLog进度）用于保存复制进度。从库上执行SHOW SLAVE STATUS可以看到当前从库复制的状态。

master.info信息

  • Master Host：主库IP
  • Master User：主库上复制使用的用户账户
  • Master Port：主库MySQL端口
  • Master_Log_File：从库的I/O线程当前正在读取的主库Binlog的文件名
  • Read_Master_Log_Pos：从库I/O线程当前读取到的位置

relay-log.info信息

  • Relay_Log_File：从库SQL线程正在读取和应用的中继日志RelayLog的文件名
  • Relay_Log_Pos：从库SQL线程当前读取并应用的中继日志RelayLog位置
  • Relay_Master_Log_File：从库SQL线程正在读取和应用的RelayLog对应于主库Binlog的文件名
  • Exec_Master_Log_Pos：中继日志RelayLog中Relay_Log_Pos位置对应于主库BinLog的位置

三种复制方式

二进制日志Binlog格式有三种，可以通过 show variables like '%binlog_format';查看

  • Statement：基于SQL语句级别的binlog，每条修改数据的SQL都会保存到BinLog里。
  • Row：基于行级别，记录每一行数据变化，也就是将每一行数据的变化都记录到Binlog里面，记录的非常详细，但是并不记录原始的SQL，在复制的时候并不会因为存储过程或触发器造成主从数据不一致的问题，但是记录日志量比Statement大很多
  • Mixed：混合Statement和Row，默认情况下采用Statement，某些情况下切换到Row模式。

MySQL复制

  • binlog_format=Statement：基于SQL语句的复制
  • binlog_format=Row：基于行的复制
  • binlog_format=Mixed：基于混合复制模式进行复制

通过set binlog_format="ROW"等命令指定binlog格式

复制的3种常见架构

一主多从

对于读取压力特别大的场景下可以通过一主多从的模式实现读写分离，把大量的实时性要求不高的读请求通过负载均衡分布到多个从库上，降低主库的读取压力

graph TD;
    A(写请求)-->B(Master)
    B-->|Replication|C(多台Slave)
    D(读请求)-->|LVS/负载均衡|C

多级复制

因为MySQL的复制是推送Binlog，所以主库的I/O压力会随着从库的增加而增长，多级复制能够解决一主多从场景下主库额外的I/O压力和网络压力。

graph TD;
    A(写请求)-->B(Master1)
    B-->|Replication|B1(Master2)
    B1-->|Replication|C(多台Slave)
    D(读请求)-->|LVS/负载均衡|C

和一主多从不同的是多级复制是在Master和slaves之间加入了一个二级主库，这样Master1接收了命令后只需要将Binlog推送给Master2，减轻了Master1推送压力。但是由于MySQL是异步复制， 经历了两次推送才到达Slaves，延迟会比一主多从的场景大。可以在Master2上选择表的引擎为BLACKHOLE来降低多级复制的延迟。BLACKHOLE引擎对于写入的数据并不会回写到磁盘，BLACKHOLE永远都是一个空表，修改操作仅仅在Binlog中记录事件

双机复制

适用于DBA做维护等需要主从切换的场景，通过双主避免了重复搭建从库的麻烦

graph TD;
    A(读写请求)-->|浮动虚拟IP|B1(Master1)
    A-->|浮动虚拟IP|B2(Master2)
    B1---|互为主从|B2
    B2---|读写分离|B1

主从切换过程

  1. 在master1上停止Slave线程，避免后续对Master2库的维护操作被实例复制到Master1库上对服务造成影响
  2. 在Master2库上停止Slave线程，开始日常维护操作
  3. 在Master2上完成维护操作之后，打开Master2哭上的Slave线程，让Master2库的数据和Master1库同步，同步完成后把应用的读写操作切换到Master2库上
  4. 最后，确认Master1库上无应用访问后，打开Master1库的Slave线程即可

通过双主复制能够减轻一主多从架构下对主库进行维护带来的额外搭建从库工作

同时双主复制可以和主从复制配合，在Master2上配置两个slave，这样可以通过从库Slave分担压力，同时在DBA维护的同时，避免重建从库的额外工作

graph TD;
    A1(读写请求)-->|浮动虚拟IP|B1(Master1)
    A2(读请求)-->|浮动虚拟IP|B2(Master2)
    B1-->|BinLog推送|B2
    B2-->|BinLog推送|C1(Slave1)
    B2-->|BinLog推送|C2(Slave2)

由于MySQL的Binlog中会记录事件初始发生的Server Id， MySQL库只应用和自己Server Id不同的Binlog日志，MySQL通过判断Binlog事件中的不同ServerId即可判断当前库事件是否发生的初始server，这样即使双主复制加上级联复制也不会出现循环复制

搭建复制

异步复制

  1. 安装相同版本的数据库

  2. 主库上设置一个复制使用的账户，该账户拥有replication slave权限

  3. 修改主数据库服务器的配置， 开启binlog， 并设置server-id。

    启动或者重启服务器

  4. 在主库上设置读锁有效，这个操作是为了确保没有数据库写操作，以便获得一个一致性的快照

  5. 得到主库上当前二进制日志名和偏移量

  6. 对主库上的数据进行数据备份，这个过程可以使用mysqldump， ibbackup等工具， 但是如果主数据库可以停止的情况下， 直接复制数据文件是最快的,将备份下来的文件放置到从服务器上

  7. 主数据库备份完毕后恢复写操作, 这个加锁的原因是为了让数据备份过程中备份的数据与看到的二进制日志和偏移量相同

  8. 修改从数据库的配置文件， 添加server-id参数， 这个server-id必须与主库不同

  9. 再从库上使用--skip-slave-start选项启动从数据库，这样不会立即启动从数据库服务上的复制进程，方便对数据库的服务进行配置

  10. 对数据库服务器做相应设置

  11. 在从库上启动slave线程

这个时候观察slave的show processlist \G，会发现有一条等待执行的是waiting for master to send event还有has read all relay log; waiting for the slave I/O thread to update it分别代表I/O、SQL线程

复制的三个线程Binlog Dump、I/O、SQL之间都是独立的，相互之间没有什么依赖关系。数据的完整性依赖与主库的Binlog的不丢失，只要主库Binlog不丢失那么就算主库宕机了，我们还可以通过Binlog把丢失的数据通过手工同步到从库上

MySQL事务/语句提交但尚未释放锁的时候，在Binlog中记录事务/语句， 对于支持事务的引擎来说每个事物提交时都需要写Binlog，对于不支持事务的引擎，每个SQL语句执行完成时，都需要写Binlog。为保证Binlog的安全，需要引入sync_binlog参数来控制Binlog刷新到磁盘的频率

半同步复制

异步复制具有主从数据库之间的延迟， 这样有可能会因为推送不及时导致Binlog的丢失

半同步复制的话，每次主库成功提交后，不立马返回给前端应用，而是等待其中一个从库也接受到Binlog事务，并且成功写入中继日志后，主库才返回Commit给客户端。这样的话，每次提交都有两个日志记录，一份在主库的Binlog上， 一份在从库的中继日志日志Relay Log上。

graph TD;
    A(clietn)-->|1.commit|B((master))
    B-->|2.Execute|C(3. write Binlog)
    C-->|4.send transaction to slave|D(5. Relay Log)
    D-->|6. send ACK|B
    B-->|7.Return client|A

其中1-3步骤中出现问题， 由于事务没有提交，从库也没接受事务的binlog日志，所以主从数据一致。如果在4过程出现问题， 导致Binlog没有传输到从库上， 主库会进行一段时间的等待（由rpl_semi_sync_master_timeout设置的时间决定)。如果这段时间内都无法成功推送到从库上，则MySQL自动调整为异步模式，事务正常返回提交给客户端

开启异步复制

  1. 查看MySQL是否支持动态增加插件

  2. 确认plugin中确实有插件

  3. 主库上安装semisync_master插件

  4. 从库上安装semisync_slave插件

  5. 分别在主库和从库上配置参数打开半同步

  1. 由于之前配置的是异步复制， 此时我们重新启动从库上的I/O线程

  2. 查看主库上的状态

    • Rpl_semi_sync_master_status:No表示打开
    • Rpl_semi_sync_master_yes_tx: 0 表示主库当前尚未有任何一个事务是通过半同步复制到从库的
    • Rpl_semi_sync_master_no_tx：0 表示当前有0个失误不是半同步模式下从库及时响应的

    如果在半同步的状态下，主库等待从库返回ACK的时间超过rpl_semi_sync_master_timeout后，会自动转为异步复制， 当Slave重新连接上主库后， 会由异步自动转为半同步

主要复制启动选项

log-slave-updates

用来配置从库上的更新操作是否写二进制日志，默认是不打开的。但是如果这个从库同时也要作为其他服务器的主库，搭建一个链式的复制，那么就需要打开这个选项。

master-connect-retry

用来设置和主库的链接丢失时重试的时间间隔，默认是60秒

read-only

read-only该参数用来设置从库只能接受超级用户的更新操作，从而限制应用程序错误的对从库的更新操作。

指定复制的数据库或者表

可以使用replicate-do-db、replicate-do-table、replicate-ignore-db、replicate-ignore-table、replicate-wild-do-table指定从主数据库复制到从数据库的数据库或者表。

slave-skip-errors

复制的过程中， 可能会遇到执行BinLog中的SQL出错， 默认情况下从库将停止复制， 不在进行同步，等待用户介入处理。但是如果加入这个参数， 将自动跳过错误号，这样会自动跳过去执行下面的语句。这个参数可以定义多个错误号，或者通过定义成all跳过全部的错误

--slave-skip-errors=[err_code, err_code2,... | all ]

如果从库主要是作为数据的备份， 那么不应该加上这个参数，因为这有可能会导致主从数据不同步， 但是仅仅作为分担主库的压力， 并且对数据的要求不严谨，可以使用这个参数

日常管理维护

查看从库状态

主从库同步维护

如果因为主库更新频繁，导致更新速度较慢，从而导致主从库之间的数据差距越来越大，最终对某些应用产生影响。此时需要定期进行主从库的数据同步，是的主从数据差距能够减小。常用的方法是负载较低的时候暂时阻塞主库的更新，强制主从数据库更新同步。

从库复制出错的处理

在某些情况下，会出现从库更新失败，这是，首先需要确定是否是从库的表与主库的不同造成的。如果是表结构不同导致的，则修改从库的表使之与主库完全相同，然后重新运行start slave

如果不是表结构不同导致的更新失败， 则需要确认手动更新时候安全， 然后忽略来自主库的更新失败语句，跳过来自主库的语句的命令为set global sql_slave_skip_counter=n(以event为单位，跳过n个单位的事务)，因为我们只用跳过错误语句，所以其中n的取值为1 或者 2， 如果来自主库的更新语句不使用auto_increment或者last_insert_id(), n值应为1，否则为2。因为使用auto_increment或者last_insert_id()的语句需要从二进制日志中取两个事件。以下例子就是在从库端模拟跳过主库的两个更新语句

log event entry exceeded max_allowed_packet的处理

如果应用中使用打的BLOG列或者字符串，因为网络无法进行传输 ，所以会报log event entry exceeded max_allowed_packet错误。可以在主从库上增加max_allowded_packet参数的大小，这个参数默认1MB， 可以按照需要修改

多主复制时的自增长变量冲突问题

大多数情况下只使用单主复制，但是某些情况下会使用多主复制，这是如果主库的表采用自动增长变量， 那么复制打哦从库的同一张表后很可能会引起主键冲突，因为系统参数auto_increment_increment和auto_increment_offset的默认值为1， 这样多台主库的自增变量列迟早会发生冲突。所以多主复制的时候需要定制auto_increment_increment和auto_increment_offset的设置，保证主从之间数据库不会有复制冲突

  • master1：auto_increment_increment=2， auto_increment_offset=1；（1,3,5,7…)
  • master2：auto_increment_increment=2，auto_increment_offset=0；（0,2,4,6…）

查看从库的复制进度

因为二进制日志中的每个语句通过设置时间戳来保证执行时间和顺序的正确性，所以每个语句执行之间都会首先设置时间戳，通过这个进程的Time就可以知道最后设置的时间戳和当前时间的差距

如何提高复制的性能

通过从库执行show processlist可以看到只有一个SQL线程在解析中继日志RelayLog。如果在写入特别大的场景，因为主库是多线程写入，但是从库只有一个SQL线程在应用日志，就容易出现从库追不上主库的情况， show slave status命令可以看Seconds_Behind_Master参数查看从库落后主库的秒数，但是这个参数不是特别精准， 只是一个预估值。

拆分减少一个从库需要数据同步的表

首先考虑一主多从的架构，然后在不同的从库上，通过设置不同的replicate-do-db、replicate-do-table、replicate-ignore-db、replicate-ignore-table、replicate-wild-do-table参数，使得不同的从库复制不同的库/表， 减少单个从库需要写入的数据。

graph TD;
    A(Master)-->B(databaseA)
    A-->C(databaseB)
    A-->D(databaseC)
    B-->E(Slave1)
    C-->F(Slave2)
    D-->G(Slave3)

由于主库需要给三个从库都发送完整的Binlog，I/O和网络的压力都会特别大，所以改为多级复制架构

  1. 让Master1给二级主库发送完整的
  2. Binlog日志二级主库打开log-salve-update开启主从模式接收主库的Binlog到RelayLog，并且开启BLACKHOLE模式，不用将数据进行落盘，降低二级主库的I/O压力。
  3. 为二级主库配置三个从库，三个从库通过配置不同replicate-do-db参数，复制不同的表
graph TD;
    A1(Master1)-->A
    A(Master2)-->B(databaseA)
    A-->C(databaseB)
    A-->D(databaseC)
    B-->E(Slave1)
    C-->F(Slave2)
    D-->G(Slave3)

这样的话能够自由拆分从库，方便把热点数据分散开来，缺点在于维护起来不够简洁，而且因为三个从库都没有完整的数据， 在主库出现意外宕机的情况处理较为麻烦

切换主从库

当主从复制时的主库出现问题时，需要将从库中的一个切换成主库，同时修改其他的从库配置，使其指向新的主库。此外还需要通知应用修改主数据库的IP地址，如果有可能，将出现故障的旧主库修复或者重置成新的从数据库

  1. 首先确保所有的从数据库都已经执行了Relay Log中的全部更新，每个从库上，执行STOP SLAVE IO_THREAD，然后检查SHOW PROCESSLIST的输出，知道看到状态是Has read all relay log

  2. 在新选为主的从数据库上，执行STOP SLAVE以停止从服务，然后执行RESET MASTER 以重置成主数据库

  3. 在其他从库上，执行STOP SLAVE以停止从服务，然后执行CHANGE MASTER TO MASTER_HOST='新主库的host'以重置主数据库，在执行START SLAVE启动复制

  4. 通知所有的客户端应用指向新主库，这样客户端发送的所有更新语法写入到新主库的二进制日志

  5. 删除新的主数据库服务器上的master.info和relay-log.info文件，否则下次重启时还会按照从库启动

  6. 最后如果旧主库可以恢复，则按照其他从库配置成新主库的从库
