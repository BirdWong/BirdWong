MySQL-数据类型的选择

发表于 2019-09-13 更新于 2019-09-14 分类于 MySQL 阅读次数： Valine：
本文字数： 4k 阅读时长 ≈ 7 分钟

CHAR与VARCHAR

CHAR和VARCHAR都是用来存储字符串，但他们保存和检索的方式不同，CHAR属于固定长度的字符类型，而VARCHAR属于可变长度类型

CAHR会删除尾部的空格

因为CHAR是固定长度，所以处理的速度比VARCHAR快得多，但是CHAR比较浪费存储空间，如果是长度变化不大并且查询速度有较高要求的应该使用CHAR

不同存储引擎的使用原则

  • MyISAM：建议使用CHAR的数据列代替可变长度的数据列
  • MEMORY：无论是使用CHAR还是VARCHAR都是使用CHAR进行处理
  • InnoDB：建议使用VARCHAR类型，因为内部存储时没有区分固定长度和可变长度，因此CHAR的性能不一定比VARCHAR好，主要的影响因素是存储量（CHAR平均占用空间多于VARCHAR）

TEXT与BLOB

两者均用于保存大类型数据

  • BLOB：保存二进制数据，比如照片

    包含类型：

    • BLOB
    • MEDIUMBLOB
    • LONGBLOB

  • TEXT：只能保存字符数据，比如文章

    包含类型

    • TEXT
    • MEDIUMTEXT
    • LONGTEXT

根据数据的类型和数据的大小选择对应的存储类型

BLOB和TEXT常见问题

  1. BLOB和TEXT在执行了大量的删除操作时会引起一些性能问题

    删除操作会在数据表中留下很大的‘空洞’，填入这些‘空洞’的记录在插入的性能上会有影响，应该定时使用OPTIMIZE TABLE对这类表进行碎片整理

  2. 查询速度慢

    可以使用合成的索引来提高文本字段的查询性能 。合成索引就是根据大文本字段的内容建立一个散列值，并把这个值存储在单独的数据列中，接下来的查找就可以通过检索散列值就能找到数据了。但是通过散列值查询只能用于精确匹配的查询（范围查询无效，例如：大于， 小于，大于等于，小于等于）。散列值可以通过MD5()/SHA1()/CRC32()生成。如果生成的散列值尾部有空格就不要放入varchar、char中存储，以免受到去除空格的影响。

    如果需要进行模糊查询，可以使用前缀索引（为前n个字符建立索引）

    前置索引进行模糊查询时， %不能在最前面，否则无法使用索引

  3. 在不必要的时候避免检索大型的BLOB或者TEXT

    例如不要select * ，除非能够确定作为约束条件的WHERE字句只会找到所需要的数据行，否则很可能会造成网络上大量传输。

  4. 把BLOB和TEXT列单独分离到单独的表中

    可以将TEXT列与BLOB列移动到第二张表中，原来列替换成固定长度的散列值，这样就可以减少主表的碎片，并且使用到固定长度的性能优势，也可以避免select *造成的大量网络资源浪费

浮点数与定点数

浮点数和定点数都可以保存含有小数部分的数据。但是如果插入一个超出精度的值时，浮点数会直接四舍五入这个数值并进行保存，定点数则会发出警告之后四舍五入保存或者报错拒绝保存。同时定点数是字符串形式保存，所以能够更加精确的保存数据。

日期类型选择

常用日期类型有：DATE, TIME,DATETIME,TIMESTAMP。

  1. 根据实际需要选择能够满足应用的最小存储的日期类型，例如：只需要保存年份， 完全可以使用一个字节的YEAR，不需要用四个字节的DATE

  2. 日过要记录年份比较久远的时间，最好使用DATETIME,而不要使用TIMESTAMP，因为TIMESTAMP的日期范围比DATETIME短得多

  3. 如果记录的日期需要让不同时区的用户使用，那么最好使用TIMESTAMP，因为日期类型中只有他能够和实际时区对应

    时间的统一可以使用：UTC 通用标准时

    应用程序将所有时间都转化为UTC 通用标准时， 使用的时候通过目前的时区转换为真正的时间
